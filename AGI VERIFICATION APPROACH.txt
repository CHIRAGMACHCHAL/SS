### Core Philosophy — Pehle Samjho, Phir Fix Karo

Hum SS AGI ki 5 layers aur unke phases ko **ek ek karke** fix karte hain — Layer 1 Phase 1.1 se shuru karke Layer 5 ke last phase tak. Sequence kabhi nahi todna. Lekin kisi bhi phase ko fix karne se pehle **poora blueprint deeply padhna** zaroori hai — sirf us phase ka section nahi, **poora** — kyunki saari layers aapas mein connected hain aur ek jagah galat fix doosri jagah tod sakta hai.

---

### Step 1 — Poora Blueprint Padho, Poora AGI Samjho

`project_knowledge_search` se **poora blueprint** padho. Sirf us phase ka section nahi — poori AGI ki working architecture samjho:

- 5 layers kya hain, kaise kaam karti hain

- Layers aapas mein kaise baat karti hain

- Response kaise generate hota hai — Layer 1 se Layer 5 tak ka poora flow

- Us phase ka **poore system mein role** kya hai

Ye samjhe bina koi fix nahi dena.

---

### Step 2 — Blueprint se Phase ka Role Quote Karo

Jo phase fix karna hai, blueprint ne uske baare mein **exact kya kaha** — wo quote karo verbatim. Assumption mat lo. Ye quote hi anchor hai jiske against saari verification hogi.

---

### Step 3 — Teen Jagah Current Code Padho

Sirf class nahi — **teen jagah** dekhna zaroori hai:

**A) Phase class khud** — andar kya likha hai, kya ho raha hai

**B) Call site** — kahan se call ho rahi hai, kya pass ho raha hai, signature match karta hai ya nahi

**C) Downstream** — is phase ka output kaun use karta hai, kaise use karta hai, wahan kuch toot to nahi raha

---

### Step 4 — Violations Dhundho, Reason Do

Har violation ke saath **reason** do — sirf "ye galat hai" nahi:

- Blueprint ne kya kaha

- Code abhi kya kar raha hai

- Dono mein conflict kyun hai

**Hamesha check karo:**

- **Labels / strings / keywords** — `"research"`, `"invention"`, `"exploratory"`, `"declarative"` — ye blueprint mein **capabilities ke examples** hain, string labels nahi. Inhe code mein likhne se AGI wo capability nahi paata. Car pe `"fly"` likhne se car nahi udti.

- **Hardcoded English templates** — `"Elaborate on:"`, `"From Vedic perspective:"`, `"ancient vs modern equivalent"` — brain mein hardcoded strings. European user French mein bole, Sanskrit mein bole — ye strings fail ho jaayengi.

- **Word length / token count based decisions** — kabhi nahi. Structure decide karta hai, size nahi.

- **Tier awareness brain mein** — `"free"`, `"jarvis"`, `"enterprise"` — brain ko tier nahi pata. Sirf config se power milti hai.

- **Dead / unused variables** — define karo to use karo, warna hatao.

- **Placeholder code** — jo sirf dikhta hai karta kuch nahi — ye nahi chahiye. Heavy, real, working code chahiye.

- **Memory Graph secondary** — blueprint kehta hai Memory Graph primary brain hai. Agar code mein secondary hai ya skip hai — violation hai.

- **Runtime errors** — variables jo exist nahi karte, wrong signatures, broken downstream connections.

---

### Step 5 — Billing Explicitly Check Karo

Har phase ke liye alag se billing check karo:

- **Agar billing variable chahiye** → reason do, variable name batao, har tier ki alag value batao — overlapping values nahi hongi

- **Agar nahi chahiye** → reason do — mandatory cognitive step hai isliye, ya tier differentiation downstream hoti hai

- **Agar pehle se hai** → billing.py verify karke confirm karo — values sahi hain, koi mismatch nahi, koi value code mein handle nahi hoti to wo fix karo

---

### Step 6 — Fix Do — Reason Ke Saath

- Pehle **kya hatana hai** — exact lines, exact reason

- Phir **kya daalna hai** — complete working code, koi placeholder nahi

- Har change ke saath **reason** — blueprint ke against verify karke

---

### Step 7 — Reverification Table

Fix ke baad blueprint se ek ek point verify karo:

| Blueprint ne kya kaha | Fix ke baad code mein kya hai | Status |

|---|---|---|

| Exact quote | Exact code behavior | ✅ / 🔴 |

### Step 8 — Purane Saare Fixed Phases Ko Verify Karo (NAYA — MANDATORY)

Ye step **har naye fix ke baad** karna zaroori hai — ek baar bhi skip nahi hoga.

**Rule:** Jab bhi koi naya phase fix karo — usse pehle fix ho chuke **saare phases aur layers** ko dobara verify karo ki naye fix ki wajah se unka kuch kharab to nahi hua.

**Kaise karo:**

- Naya fix batane ke baad — seedha green flag mat do

- Pehle scroll karke ya code padh ke confirm karo ki Layer 1, Layer 2, aur saare pehle fix hue phases ke:

  - Class signatures intact hain

  - Call sites sahi hain

  - Output structure nahi toot

  - Koi naya dead variable introduce nahi hua

  - Koi pehle fixed violation dobara nahi aayi

**Example:**

Layer 1 fix ho chuki hai. Hum Layer 2 fix kar rahe hain.

Layer 2 ka fix batane ke baad — pehle Layer 1 ke saare phases verify karo ki Layer 2 fix ne unhe toda to nahi.

Fir Layer 2 ko green flag do.

Jab Layer 3 fix karein — Layer 1 + Layer 2 dono verify karo.

Jab Layer 4 ka Phase 2.1 fix karein — Layer 1, Layer 2, Layer 3 + Layer 4 ke pehle fix hue sub-phases sab verify karo.

**Agar kuch toot gaya ho** — pehle wo fix karo, phir naye phase ko green flag do. Partial green flag kabhi nahi.

---

### Step 9 — Car Wali Galti Check (NAYA — MANDATORY, HAR FIX KE BAAD)

Ye check **alag se explicitly** karna hai — Step 4 ke saath merge mat karo. Ye ek dedicated verification step hai.

**Car Wali Galti kya hai:**

Car pe "FLY" likhne se car fly nahi karti. Usi tarah code mein `"research"`, `"invention"`, `"factual"`, `"conceptual"` — ye strings likhne se AGI wo capability nahi paata. Ye sirf labels hain, capabilities nahi.

**Har fix ke baad explicitly check karo:**

1. **User query pe koi English keyword match nahi hona chahiye**

   - `question.lower()` pe koi `if "how" in q` jaisa code nahi

   - Hindi/Sanskrit/French mein same question aaye to bhi same result milna chahiye

   - spaCy POS/NER/dep_ se linguistic signals lo — language se upar hain ye

2. **Layer 1 ke output labels ko downstream string match nahi karna chahiye**

   - `intent_type == "research"` — ye car wali galti hai

   - `layer1_bundle.get("required_depth")` numeric index mein convert karo — phir numeric comparison karo

   - `sub_goals` count use karo — string value nahi

3. **Blueprint ke capability words code mein string constants nahi banenge**

   - Blueprint mein likha hai `"factual"`, `"conceptual"` — ye examples hain routing ke liye, hardcoded if-else labels nahi

   - Real signals: `depth_idx >= 3`, `entity_count >= 2`, `verb_count > noun_count` — ye real computed signals hain

4. **Internal routing categories alag hain user-facing strings se**

   - Phase 3.0 jo `"factual"/"conceptual"/"mixed"` return karta hai — ye Phase 3.0 ka internal computed output hai, user query keywords nahi

   - Ye categories spaCy + depth_idx + sub_goals count se compute hoti hain — ye sahi hai

   - Galat hota agar ye `"how" in question` se decide hota

**Check format — explicitly likho:**

```

Car Wali Galti Check:

- User query pe English keyword match: ✅ Nahi hai / 🔴 Hai — line X

- Layer 1 labels string match: ✅ Nahi hai / 🔴 Hai — line X  

- Blueprint words hardcoded: ✅ Nahi hai / 🔴 Hai — line X

- spaCy/numeric signals use: ✅ Ho raha hai / 🔴 Nahi ho raha

```

---                                                                                                                                                     ### **Step 10 — Capability Label → Real Power Conversion (NEW MANDATORY STEP)**

**Blueprint mein aapne jo capabilities likhi thi** (research, decode ancient tech, analysis, invention, ancient_modern_blend, temple_geometry_analyzer, viman reconstruction, scripture scientific decoding, virtual simulation, Chanakya-style decision making, new technology invention etc.) — yeh **sirf string labels nahi** the.

**ChatGPT ne galti se inko hardcoded strings bana diya** (jaise `if intent == "research"` ya `structure == "vedic-scientific-report"`).

**Car pe “FLY” likhne se car udti nahi** — isi tarah string likhne se AGI research ya decode nahi karne lagta.

**Is step ka kaam:** 1. Har phase mein saare capability strings dhundho (research, invention, decode, analysis, ancient_tech, blend, simulation, etc.) 2. Un strings ko **completely hatao** 3. Unki jagah **real capability implement karo**:    - Deep reasoning + multi-step chain    - Tool calling (ancient_tech_decoder, virtual_simulation, scripture_cross_reference, temple_geometry_analyzer, ancient_modern_blend)    - Agency engines (GoalFormation → PlanSynthesis → ActionSelection)    - Memory Graph + Vedic cross-reference injection    - Meta-cognition + contradiction resolution    - Phase 6 training hooks (sirf Jarvis mein)

4. Verify karo ki ab AGI **actually** wo kaam kar raha hai, na ki sirf label match kar raha hai.

**Verification questions har fix ke baad:** - Kya yeh capability ab string label se aage real engine ban chuki hai? - Kya Hindi/Sanskrit/French query pe bhi same real power activate hoti hai? - Kya public tier mein yeh capability safely blocked hai aur Jarvis mein fully enabled hai? - Kya koi naya tool/engine add karna pada (agar haan to uska naam aur file mention karo)

Agar kisi capability ko real power nahi mila → us phase ko **GREEN FLAG mat do** — pehle real implementation complete karo.

**Note:** Agar real power dene ke liye naya engine/tool chahiye (jaise CapabilityRealizer class) to usko bana sakte hain — lekin brain ko tier pata nahi chalna chahiye.

- Sab sahi → **GREEN FLAG** — agle phase pe chalo

- Kuch baaki → clearly batao kya karna hai, tab tak green flag nahi

**Green flag tab hi dena jab:**

1. Current phase ka code sahi ho

2. Purane saare fixed phases intact hon (Step 8)

3. Car wali galti kahi nahi ho (Step 9)

4. Koi partial fix pending na ho

### Core Rules — Kabhi Nahi Bhoolna

> **Labels nahi, capabilities do.** Blueprint ke keywords — examples hain. Inhe string banane se AGI kuch nahi karega.

> **Concepts nikal ke aate hain** — Memory Graph + spaCy se emerge hote hain. Hardcode nahi hote.

> **Brain ko tier nahi pata** — sirf config se power milti hai. Brain clean rehta hai.

> **Language agnostic** — Hindi, French, Sanskrit, English — sab chalega. spaCy dep_/pos_/entities language se upar hain.

> **Call site bhi fix karo** — class fix karna kaafi nahi. Calling convention bhi match karni chahiye.

> **Downstream bhi check karo** — is phase ka output todne se aage kuch toot sakta hai.

> **Koi dead code nahi** — define karo to use karo.

> **Koi placeholder nahi** — real, heavy, working code chahiye.

> **Sequence follow karo** — sirf current phase fix karo. Blueprint poora padho lekin fix sirf sequence mein.

> **Purana kharab mat karo** — naya fix batane se pehle hamesha verify karo ki Layer 1 se abhi tak fix hue saare phases intact hain. Ek bhi cheez toot gayi to pehle wo theek karo.

> **Car wali galti zero tolerance** — har fix ke baad explicitly check karo. Ek bhi English keyword user query pe match nahi hona chahiye. spaCy + numeric signals — yahi sahi tarika hai.
